# Issue: [标题]

## Goal

让 Pi 的 Bash 工具在每次产生增量输出时，立即通过原生 Grok Pager 工具卡显示，而非仅在工具结束后显示。

## 背景/问题

`bash` 运行会持续写入 stdout 的命令（例如每秒输出一个随机数）时，Pi-Grok 当前不会在终端实时显示这些输出。需要确认 Pi RPC 的工具更新事件是否被适配器丢弃、覆盖或被 Pager 忽略，并以最小改动保留原生工具卡渲染路径。

## 验收标准 (Acceptance Criteria)

- [ ] WHEN Bash 工具在一次调用中产生多次增量 stdout，系统 SHALL 为同一工具调用持续发布可见更新。
- [ ] WHERE Pi RPC 事件经过 `pi-grok-adapter`，系统 SHALL 保留事件顺序和工具调用 ID。
- [ ] IF 工具完成，THEN 最终工具结果仍按现有原生 Grok 工具卡语义渲染。

## Notes

- 实测 `pi --mode rpc`：三次、每秒一次的 `tick` 输出只在 5.099 秒收到最终 `response`，中间没有 Bash 输出事件。
- `PiAcpAgent::execute_bash()` 在 `crates/codegen/pi-grok-adapter/src/pi_adapter.rs` 中等待该 response，因而只能在完成时发送一次 ACP `ToolCallUpdate`。
- Pi 核心的 `AgentSession.executeBash()` 实际支持 `onChunk`；interactive mode 将该回调接到 `BashExecutionComponent.appendOutput()`。但 RPC 的 `case "bash"` 显式传入 `undefined`，没有把该回调转成 JSONL 事件。
- Pager tracker 已支持执行工具的增量输出；阻塞点在 Pi RPC 的 direct Bash 协议，不是 Pager renderer。
- 官方 extension API 可替换 `user_bash` 的执行 operations，但不能把其 stdout 作为 native tool-card 增量事件经 RPC 发送。
