# Issue: 以 Pi 为准同步项目可信状态

## Goal

将 Pi 已解析的项目可信状态同步给 Grok Pager；Pi 是唯一判定源，Grok 只显示和消费该状态，不再执行第二套 trust 决策。

## 背景/问题

Pi 在启动时依据其 `ProjectTrustStore`、CLI override 和项目级资源解析 project trust。Pager 有自身 `project_trusted` UI 状态承载点，但 Pi ACP profile 尚未同步 Pi 的判定结果，导致外部模式可能出现状态缺失或与 Pi 实际加载资源不一致。

## 验收标准 (Acceptance Criteria)

- [ ] WHEN Pi 完成项目可信判定，系统 SHALL 将布尔状态经 Pi RPC 和 adapter 同步到 Pager。
- [ ] WHERE Pi 解析项目级资源，系统 SHALL 保持 Pi 为唯一 trust authority，Pager 不得提示、覆盖或重新计算该判定。
- [ ] IF Pi session/CWD 切换导致可信状态改变，THEN Pager SHALL 接收并使用最新 Pi 状态。 

## 实施阶段

### Phase 1: 规划和准备
- [x] 调查 Pi trust 状态可用的 RPC/extension 接缝。
- [x] 调查 Pager external-profile 的 native 状态承载和展示接缝。
- [x] 确定只同步、不重复判定的最小协议。

## Notes

- `get_state` 不含 project trust。使用 `ctx.isProjectTrusted()` 的 CLI extension 避免扩展 Pi RPC；该值包含 CLI override 与临时决策。
- `grok-pi` 在启动 Pi 前安全创建临时 `.ts` extension 并以 `--extension` 注入，确保 bridge 不依赖未受信任项目的 `.pi/extensions`。
- extension 在 `session_start` 以固定 `project-trust` key 调用 `ctx.ui.setStatus()`；现有 adapter/Pager 已将其映射至原生 status bar。
- Pi RPC 的独立探针确认该 extension 发出 `Pi project: trusted`。
- `cargo test -p xai-grok-pager-bin --bin grok-pi` 受现有无关编译错误阻塞：`crates/codegen/xai-grok-pager/src/app/dispatch/session/modal.rs:92` 存在可变借用冲突（E0499）。
