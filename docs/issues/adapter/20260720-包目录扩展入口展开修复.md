# Issue: 包目录扩展入口展开修复

## Symptom

`./target/debug/grok-pi` 在 Pi RPC bootstrap 退出。stderr 显示 `pi-autoresearch/extensions/pi-autoresearch/compaction.ts`、`hooks.ts`、`jsonl.ts` 等内部模块被逐个加载为 Pi extension factory，并报 `Extension does not export a valid factory function`。

## Root Cause

Rust `package_resource_paths()` 把 package manifest 的目录条目（例如 `"./extensions"`）交给通用递归 `discover_resource_path()`。对于 Extensions，这会枚举所有 `.ts/.js` 文件。Pi 0.80.10 的 `collectFilesFromPaths()` 则针对 extensions 调用 `collectAutoExtensionEntries()`：目录只贡献 manifest 声明入口或 `index.ts/index.js`，不递归投影内部模块。

## Scope

- 在 `pi_resource_config.rs` 为 package manifest entries 使用资源类型专用展开。
- Extensions 目录复用现有 `auto_extension_paths()`；其他类型保留各自递归发现语义。
- 添加 nested extension module 回归测试。
- 验证实际 `pi-autoresearch` catalog 只产生 `index.ts` extension，且不再生成 stderr 中的内部模块路径。

## Acceptance Criteria

- [x] manifest `extensions: ["./extensions"]` 且子包有 `index.ts` + 内部模块时，仅 catalog `index.ts`。
- [x] skills/prompts/themes 的目录 manifest entries 继续递归发现合法资源。
- [x] `pi-autoresearch` 不再将 `compaction.ts/hooks.ts/jsonl.ts/paths.ts/shortcuts.ts` 放入 launch plan。
- [x] 定向 tests、policy tests 和 grok-pi binary check 通过。

## Verification

- Regression tests cover both a nested extension package (`extensions/example/index.ts`) and an extension directory with direct `index.ts`, implementation module and `*.test.ts` sibling.
- `cargo test -p xai-grok-pager --lib pi_resource_config`：17 passed。
- `cargo test -p xai-grok-pager --lib pi_resource_policy`：21 passed。
- `cargo test -p xai-grok-pager --lib pi_config`：11 passed。
- `cargo check -p xai-grok-pager-bin --bin grok-pi`：通过（仅既有 dead-code warnings）。
- Real PTY bootstrap of freshly built `./target/debug/grok-pi` reached Pi host/TUI initialization without `failed to bootstrap Pi RPC state` or `Extension does not export a valid factory function`.
