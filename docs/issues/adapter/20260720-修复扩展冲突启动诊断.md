# Issue: 修复扩展冲突启动诊断

## Goal

在不修改 Pi 源码的前提下，让 grok-pi 对 Pi RPC 启动失败提供可操作的扩展冲突诊断，并设计由宿主实施资源准入策略的最小边界。

## 背景/问题

Pi RPC 子进程启动期间的 stderr 目前只进入 tracing，bootstrap 仅收到 `Pi RPC stdout closed`。已用 `grok-pi --extension ~/.pi/agent/npm/node_modules/pi-tool-display/index.ts` 复现相同失败；`pi-tool-display` 会重注册 read/grep/find/ls/bash/edit/write，并为后续注册 monkey-patch `pi.registerTool`，与 grok-pi 的 RPC 工具投影不兼容。

## 验收标准 (Acceptance Criteria)

- [ ] WHEN Pi RPC 在 bootstrap 前退出，系统 SHALL 显示退出码与 stderr 尾部诊断，而非仅显示 stdout 关闭。
- [ ] WHERE 资源属于 Pi-TUI renderer 且宿主为 grok-pi RPC，策略 SHALL 默认禁用该资源并说明原因。
- [ ] IF 用户显式允许一个被策略阻止的资源，THEN 系统 SHALL 允许启动并保留清晰风险提示。

## 关键决策

| 决策 | 理由 |
|------|------|
| 宿主策略而非修改 Pi | Pi 是外部依赖；`--no-*` + 显式资源是官方接缝，无需 fork |
| 传具体资源路径而非包源 | `--extension <npm:pkg>` 会展开包内全部资源，绕过细粒度过滤 |
| 默认黑名单 + 启发式 | pi-tool-display 重注册 7 个内置工具并 monkey-patch registerTool，必须阻止；名称启发式覆盖未知 renderer |
| stderr 环形缓冲区 32 行 | 足够显示扩展加载错误，不占过多内存 |

## Notes

- 2026-07-20：PTY 下默认 grok-pi 与 --no-extensions 均可启动；显式注入 pi-tool-display 后稳定失败，说明现有泛化错误掩盖了扩展加载/注册失败。
- Pi 0.80.10 支持 `--no-extensions/--no-skills/--no-prompt-templates/--no-themes` 加显式重复资源参数；禁用自动发现后，显式资源仍会加载。
- 重要限制：`--extension <npm/git/local package source>` 会按包 manifest 展开所有资源类型，不能作为细粒度 package allowlist。要实现类型级准入，宿主目录必须使用现有 catalog 对包资源展开为具体条目，再按策略显式传入；Pi 仍是唯一的扩展执行/注册者。
- 未信任项目资源与项目 settings 仍由 Pi trust 语义裁决；策略不能绕过 trust。若要完全控制上下文，另需评估 `--no-context-files`。
