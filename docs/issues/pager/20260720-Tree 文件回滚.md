# Issue: Tree 文件回滚

## Goal

为 `grok-pi` 提供默认关闭、由 F2 显式启用的 **Tree 文件回滚**：用户在当前 Pi SessionTree 的 active path 上选择祖先节点，恢复该节点完成后的文件状态。

功能只回滚由 Pi 内置 `write` / `edit` 实际造成的普通文件内容变化；文件可以位于会话 cwd 内或 cwd 外。即使工具因“写盘后的 abort/错误”最终返回失败，只要磁盘已变化，也必须记录。它不改变 Pi 对话、session tree、active leaf，也不执行任何 Git 写操作。

---

## 用户规格（优先级最高）

以下是本功能的主规格。Cursor、Codex、Claude 等外部产品只能用于检查实现风险，不能改变或扩展这些规则；发现逻辑矛盾或无法满足的前提时必须先向用户说明。

1. **路径范围**：从当前 active leaf 回溯至 root 的单一路径中选择目标节点；禁止跨分支。
2. **目标语义**：目标为 `X5` 时，保留 `X5` 自身产生的 write/edit 改动，只撤销 `X5` 的后代节点产生的改动；恢复为 `X5` 完成后的文件状态。
3. **文件范围**：仅回滚 Pi 内置 `write` 与 `edit` 实际造成的文件变化；相对路径与 cwd 外绝对路径均应覆盖。目标之后由 `write` 新建的受管文件应删除。工具结果是否为 error 不能代替磁盘变化事实。
4. **不改变会话**：文件回滚后 active leaf 仍保持当前节点（例如 `X10`），不调用 `navigateTree()`，不截断/重写对话历史。
5. **冲突规则**：如果某一受管文件自最后一次受管写入后被外部修改、外部创建或外部删除，整体拒绝；不得部分恢复。
6. **默认关闭**：Pi 没有原生该能力。功能必须默认关闭，并通过 `grok-pi` F2 设置显式启用。
7. **恢复会话**：`--continue`、`/resume` 或其他恢复的会话不能因“不是本进程新建”而一概失去文件恢复能力；只要路径和可验证的恢复状态明确，系统应尽可能帮助恢复。无法证明的部分必须明确拒绝，而非猜测或覆盖。
8. **当前激活会话**：回滚操作只可作用于当前激活的 Pi 会话。

## 现状事实：write/edit 是内存还是持久化？

### 文件本体

Pi 内置工具对文件的修改是**直接持久化到本地文件系统**，不是先写入 Pi 内存、等待会话结束再落盘：

| 工具 | 实现事实 | 证据 |
|---|---|---|
| `write` | 在每个路径的 mutation queue 内创建父目录，然后直接调用 `fs.promises.writeFile(absolutePath, content, "utf-8")`。 | `pi-main/packages/coding-agent/src/core/tools/write.ts` |
| `edit` | 在 mutation queue 内读取文件、计算精确替换，随后直接调用 `fs.promises.writeFile(absolutePath, finalContent, "utf-8")`。 | `pi-main/packages/coding-agent/src/core/tools/edit.ts` |
| 并发 | Pi 对同一路径的工具修改通过 `withFileMutationQueue()` 串行化；不同路径仍可并发。 | `pi-main/packages/coding-agent/src/core/tools/file-mutation-queue.ts` |

因此，**文件本身已经落盘；Pi session 不是文件恢复存储**。

### Pi session / Tree

Pi 会将用户、助手和工具结果写入 session JSONL，并由 `SessionManager` 以 entry tree 维护 parentId/leafId。`getBranch()` 能给出 active leaf 到 root 的单一路径，`navigateTree()` 可切换 Pi 上下文，但不会恢复磁盘文件。

Pi 历史通常保留：

- `write` 的 `path` 与写入内容；
- `edit` 的 `path` 与 `oldText/newText`；
- 工具调用/结果及其对应 tree entry。

但 Pi 历史**不保证保留每次写入前的完整文件内容**。尤其是：

- `write` 覆盖一个已有文件时，原文件完整内容不在 write 参数或默认 tool result 中；
- 一连串 `edit` 的 `oldText` 仅覆盖被替换片段，不能可靠重建未涉及的其余文件内容；
- Pi compaction 会追加 compaction entry，但不会删除原 JSONL tree entries；因此 compaction 本身不是历史丢失原因。真正的缺口是旧会话未捕获 preimage、损坏/截断会话、工具实现变化，或非 Pi 写入。

结论：Pi 的 session/tree 可提供“何时、哪个节点、哪个路径被工具操作”的语义锚点，但**不能单独作为可靠的文件快照库**。

## 设计结论：注入的 Pi extension 拥有持久 checkpoint journal

受控 Pi extension 必须拥有快照、journal 状态机、路径锁、目标状态计算与实际恢复；组合 binary 只负责注入 extension 和提供私有状态根，adapter 只转发版本化 bridge 请求/响应，Pager 只使用原生 Tree/确认/toast 表面。这样既不改 Pi 源码，也不让 adapter 拥有 UI 或复制一套文件恢复算法。

### 为什么不是纯内存状态

纯内存只支持同一进程，无法处理用户要求的恢复会话场景，也会在 Pager/Pi 子进程异常退出后失去恢复能力。因此“Grok 侧维护状态”应解释为：

```text
Pager：原生 Tree / preview / confirm / toast
  │
adapter：ACP ext_method ↔ 隐藏 bridge command；不读写目标文件
  │
Pi extension：工具包装、WAL、blob、tree binding、preview、restore
  │
私有状态根：session journal + content-addressed byte blobs
```

journal 必须使用 `$GROK_HOME/pi-file-rollback/`（默认 `~/.grok/pi-file-rollback/`）作为私有状态根，且不能放入被回滚目标的项目工作区。组合 binary 创建 process-unique control directory，并通过 child-only env 把路径交给 extension，同时把同一路径显式传给 `PiAgent`；不得使用进程全局 env 让 adapter 再读取。持久 session 使用 canonical session file + session id 派生 sidecar identity；`--no-session` 没有可恢复的 session tree 身份，只允许进程内临时捕获，不承诺重启后 resume。外部路径的 blob 同样存入该状态根，不写入外部文件的相邻目录。

### 最小数据契约

每一个实际受管 mutation 都记录一条不可变 operation：

```text
RollbackJournalHeader
  version
  piSessionId
  canonicalSessionFile?
  sessionHeaderDigest?
  origin = new | continued | resumed | forked | ephemeral
  captureBoundaryEntryId?     # 本 journal 开始完整捕获时的 leaf
  createdByGrokPi

MutationRecord
  sequence
  operationId
  piSessionId
  treeEntryId?                # 成功绑定后必须是对应 toolResult entry
  toolCallId
  tool = write | edit
  canonicalPath               # cwd 外允许；记录稳定的实际目标
  before = absent | blob(hash)
  after = absent | blob(hash)
  state = prepared | committed | unbound | reconciled
  toolReportedError
  preparedAt
  committedAt?

RollbackTransaction
  transactionId
  targetEntryId
  sourceLeafId
  plannedPaths
  state = prepared | committed | compensating | failed
  createdAt
```

`before` 与 `after` 都是恢复的权威数据，内容按原始 bytes 保存而不是 UTF-8 字符串：

- 文件原先存在：保存完整 preimage blob；
- 文件原先不存在：保存 `absent`；目标状态为 absent 时删除该文件；
- 写盘前先 durable `prepared`，写盘并读取实际 bytes 后再 durable `committed`；
- 工具可能在 `fs.writeFile()` 成功后因 abort 检查而报错，此时仍提交实际 mutation，并设置 `toolReportedError=true`；
- 崩溃遗留的 `prepared/unbound` 记录在 resume 时通过当前文件、toolCallId 与 session entries 对账；无法证明时阻止相关回滚。

### 记录时机、工具包装与原子性

不能只在 `tool_call` / `tool_result` event 中读文件：Pi 可并行执行工具，且内置 mutation queue 在 tool `execute()` 返回时已经释放；另一个同路径调用可能在 `tool_result` post-read 前写入。正确方案是：

1. rollback extension 必须在所有用户 extension 之后注入，但不能在 factory 初始化时立刻覆盖工具；它在 `session_start` 先通过 `getAllTools()` 检查 `write/edit` source 必须仍是 Pi builtin。若被用户 extension/custom tool 同名覆盖，则该工具 fail-closed，不得覆盖用户实现；
2. 检查通过后才调用 `registerTool()`，使用 Pi 公共导出的 `createWriteToolDefinition()` / `createEditToolDefinition()` 和 `withFileMutationQueue()` 构造同语义包装工具。Pi 的“extension tool first registration wins”继续保证用户先注册的同名工具优先；通过 per-call context 将 `toolCallId` 带入自定义 operations；
3. preimage、WAL prepare、真实 write、after blob 与 WAL commit 全部发生在同一路径 mutation queue 临界区内；任何快照/权限/空间错误都在写盘前阻止工具，绝不允许“写了但未受管”；
4. 回滚只在 Pi idle 时启动，先取得全局 rollback lock，再按稳定顺序取得所有受管路径锁，完整预检并再次校验当前 bytes；
5. `write` 的 mkdir 与文件写入、以及 `edit` 的 access/read/compute/write 必须完整保留在同一个 outer mutation queue 临界区；不得让底层 native definition 再次获取同一路径 queue，避免自锁；
6. 预检冲突保证零写入。进入 commit 后使用 durable `RollbackTransaction` 与补偿记录，使进程崩溃或中途 I/O 失败可在下次启动继续或补偿。

外部进程不受 grok-pi 的锁控制，因此不能宣称数学意义上的跨进程事务原子性；实现提供“内部串行化 + commit 前复验 + crash-recoverable WAL”，并将可检测外部竞态作为冲突拒绝。

不以 Git 状态、Git stage 或 Git checkout 作为恢复媒介。回滚结果只是普通未暂存的工作区文件变化。

## Tree 与文件操作的绑定

### 当前分支限定

操作只能以当前 active leaf 为起点：

```text
current leaf X10
  └─ parent
      └─ ...
          └─ X5  ← 用户选择的 ancestor target
              └─ ...
                  └─ root
```

执行前由 Pi `get_tree` / 当前 branch 做以下验证：

1. 当前 Pager agent 对应当前 active Pi session；
2. target entry 存在；
3. target 位于 current leaf → root 的 path；
4. journal operation 的 `treeEntryId` 位于 target **之后**的同一路径；
5. 任何不在路径上的 operation 都不能参与恢复。

这保证不会跨分支，也不会依赖 Tree UI 的视觉压缩/过滤结果来决定安全边界。

### X5 边界

选择 `X5` 时：

```text
保留：root ... X5 自身完成时已经存在的文件状态
撤销：X5 的严格后代节点关联的受管 write/edit operations
不变：Pi leaf / Pi message context / 其他分支
```

### Tree entry 绑定语义

一个 toolCallId 同时出现在 assistant tool-call 内容和后续 toolResult message 中。文件 mutation 统一绑定到对应 **toolResult entry**：该节点代表工具执行结果已经形成，选择它作为 X5 时保留该次改动；选择其 assistant 父节点时撤销该次改动。

Pi 在 `message_end` extension event 返回后才 append message entry，但 `turn_end` / `agent_settled` 时 toolResult entries 已持久化。正确过程是：

1. 工具包装先以 `toolCallId` 提交 `unbound` mutation；
2. extension 在 `turn_end`（并在 resume reconciliation 再执行一次）通过只读 `ctx.sessionManager.getEntries()` 将 toolResult.toolCallId 绑定到 entry id；
3. adapter 不负责猜测或持久化该映射；
4. 无 toolResult entry 的已写盘记录保持 `unbound`，阻止覆盖相关路径的 Tree 回滚。

若当前磁盘状态包含当前 active path 之外分支的受管 mutation，则视为跨分支 provenance 冲突；不得以当前分支回滚覆盖其他分支产生的文件状态。

## 恢复 / continue 的智能策略

用户要求“只要路径明确就尽可能恢复”。这可行，但必须区分证据等级，避免把当前内容当成历史 preimage。

| 会话来源 | 可恢复级别 | 条件 | 行为 |
|---|---|---|---|
| 本进程新建会话，启动时已启用 | 完整 | journal 从空 tree/root 开始捕获，mutation 均有 before/after + tree binding | 支持受管祖先目标的完整回滚。 |
| 重新打开有完整 journal 的 session | 完整 | session id、canonical session file/header digest、tree binding 与 blobs 均匹配 | resume/continue 不构成拒绝理由。 |
| 恢复旧 session 后才首次启用 | 边界后完整 | header 记录启动时 `captureBoundaryEntryId` | 默认只允许恢复到 capture boundary 或其后可证明节点。 |
| 无 journal，但 Pi 历史可重放出目标和当前完整 bytes | 有限 | 目标状态必须有完整 write 基线，之后所有 write/edit 可重放，且重放后的 current digest 等于磁盘 | 可标记为“从 Pi 历史重建”；仅知道路径不够。 |
| 缺少完整基线/after proof | 不可恢复 | 首次观测是覆盖 write（无法证明此前存在性）、只有 edit 链、记录缺失或当前 bytes 不匹配 | 整体拒绝；不能猜测或用当前文件反推。 |

智能并不意味着静默做风险推断。它意味着：

- 合并 journal before/after blobs 与 Pi session 中明确的 `write` 内容、`edit` 输入；
- 为每个路径证明“目标 bytes”和“当前受管 bytes”；
- 只对证据链完整的路径生成恢复操作；
- 使用全量原子语义：只要候选范围内有不可证明路径或冲突，整体拒绝，并显示哪些路径阻塞。

成功回滚本身也写入 `RollbackTransaction`。后续再次回滚时不能简单地“逆序重放 target 后的原 mutation”；必须从保存的 before/after blobs 直接计算目标节点文件状态，再将当前已验证状态转换到目标状态。这样才能支持 active leaf 不变、重复回滚和回滚后继续写入。

不提供“只恢复可证明路径”的降级模式；它会产生部分文件恢复，无法满足目标节点对应完整文件状态的语义。

## 冲突与安全检查

回滚预览针对每个路径比较当前文件状态与 journal 记录的 `after` blob digest：

| 记录的 after 状态 | 当前状态 | 结论 |
|---|---|---|
| 文件内容 digest | 相同 digest | 可安全恢复下一层 preimage。 |
| 文件内容 digest | 文件内容不同 | 外部修改冲突，整体拒绝。 |
| 文件内容 digest | 文件不存在 | 外部删除冲突，整体拒绝。 |
| absent（受管操作原本创建文件） | 文件仍存在，且 digest 匹配 | 可安全删除。 |
| absent | 文件不存在 | 已处于目标状态，可视为无操作。 |
| absent | 文件存在但 digest 不匹配 | 外部创建/修改冲突，整体拒绝。 |

额外边界：

- 仅处理普通文件；目录、FIFO、socket、device 一律拒绝；
- 路径可以在 cwd 外；相对路径按 extension context cwd 解析，父目录 realpath 与最终 canonical target identity 一并记录；
- 最终目标若是 symlink，一律在写盘前阻止该次 `write/edit`。父目录中的 symlink 由 realpath 固定为 canonical parent，并在回滚时要求仍解析到同一目标；
- 只恢复文件 bytes；`write` 自动创建的父目录不删除；
- journal 目录权限必须为 0700、metadata/blob 文件为 0600，避免 cwd 外敏感文件快照泄漏；内容以 hash 去重；本 Issue 必须定义显式清理命令、保留期限和 quota，不能依赖 Grok/Pi session picker 删除自动清理 sidecar（当前没有该删除事件桥）；
- 无法 durable 保存 preimage（权限、空间、损坏等）时阻止工具写入；
- 回滚开始时会话必须 idle；运行中有 write/edit 时拒绝，不与模型工具争抢文件。

## F2 设计

功能必须默认关闭。F2 设置添加一个仅 grok-pi external profile 有意义的 Agent 项：

```text
Pi tree file rollback     Off (default)
  Track write/edit preimages and allow file-only rollback from SessionTree.
  Takes effect for new grok-pi sessions.
```

语义：

- F2 改动写入 grok-pi/Pager 配置；
- **只影响之后启动的新 grok-pi Pi 子进程**，不会对已运行会话补造不存在的 preimage；
- 启动时开关为 On 才注入 checkpoint extension 并创建/继续 journal；恢复旧 session 时记录 capture boundary；
- Off 时不捕获、不显示 Tree 回滚动作、也不执行已有 journal；关闭不会删除 journal，重新 On 后可继续验证使用；
- normal Grok profile 不暴露 Pi 特有行为。当前 Settings registry 没有 external-profile 可见性过滤，必须先补该窄接缝；否则 Pi 设置会泄漏到普通 Grok F2；
- `--no-tools` / `--no-builtin-tools` 时不得注入包装工具；`--tools` / `--exclude-tools` 只允许包装 CLI 最终仍允许的 builtin write/edit，不能借 rollback feature 重新启用被 CLI 禁用的工具。

当前工作区已有另一个进行中的 F2 Pi 内置工具设置变更，正修改 `UiConfig`、Settings registry/defs、setter 和 grok-pi 启动期注入路径。Tree 文件回滚必须等待该改动收敛后在同一配置链上最小追加，避免覆盖并发工作。

## 原生 UI 方案（不新增可视化）

按用户要求，本阶段不创建图表、网页或独立可视化。

复用已有 Pager `SessionTree` modal：

1. 只有开关开启、会话/路径满足资格时，选中当前 path 祖先节点显示 `Rollback files to this point`；
2. 该操作先显示原生 preview/confirm：目标节点、将恢复/删除的路径数、冲突数、不会改变对话/active leaf 的提示；
3. 有冲突、未绑定 snapshot 或证据链缺失时，只显示拒绝原因，不提供强制覆盖；
4. 确认后调用 adapter 的 stateful bridge，使用 Pager toast / 现有错误表面报告结果；
5. 不复用 Grok Shell `/rewind` 的 `All / ConversationOnly / FilesOnly` modal，因为其数据所有权和语义不同。

## 外部参考的核对结论（非规格）

- Cursor：checkpoint 是 agent session 内、本地、与 Git 分离的文件快照；这支持“受管工具 + 进程外持久 journal”的实现方向。
- Claude Code Agent SDK：文件 checkpoint 明确追踪受支持的写入类工具；这支持仅处理 `write/edit`，不错误声称 Bash 写入也可恢复。
- Codex 社区反馈：回滚不应隐式 stage 或修改 Git 索引；本设计禁止 Git 写操作。

这些资料没有推翻用户规则。已决策：最终目标符号链接在写盘前拒绝；恢复/continue 中任一候选路径缺少完整证据时整体拒绝。

## 验收标准

- [ ] F2 默认 Off；启用只对后续新 grok-pi 会话开始捕获生效。
- [ ] 新会话中 Pi 内置 `write/edit` 对 cwd 内外普通文件造成的实际 mutation（包括写盘后报错）均记录 durable before/after byte blobs、tool call 与 toolResult entry binding。
- [ ] 从 X10 选择祖先 X5 时，仅撤销严格后代操作；X5 自身改动、active leaf、对话和分支均保持不变。
- [ ] 目标之后由 Pi `write` 创建的受管文件被删除；原有文件恢复其历史内容。
- [ ] 任一外部冲突、非祖先 target、非当前 session、运行中会话、无完整证据路径均不产生文件写入。
- [ ] `--continue` / `/resume` 在找到匹配的完整 journal 时可恢复；首次启用的旧 session 以 capture boundary 限定可恢复范围；无证据路径明确报告不可恢复原因，不猜测。

- [ ] 不执行 Git stage/restore/checkout/reset，不改 Pi 源码，不让 adapter 拥有 UI。

## Review 结论（2026-07-20）

| 级别 | 发现 | 处理结论 |
|---|---|---|
| BLOCKER | `tool_call/tool_result` 文件读取不在 Pi mutation queue 内，并行同路径工具会产生错误快照。 | 必须包装 builtin write/edit operations，在原 queue 临界区内记录。 |
| BLOCKER | Pi 在 `writeFile()` 后再次检查 abort；工具可返回 error 但文件已改变。 | 以实际 before/after bytes 为准，不能以 `isError` 决定是否记录。 |
| BLOCKER | active leaf 不变时，成功回滚后再继续写入或再次回滚，简单逆序 operation 模型失效。 | 保存 before+after blobs，并将回滚本身作为 transaction；直接计算目标状态。 |
| HIGH | adapter 无法从隐藏 extension command 直接取得结构化 preview payload。 | 定义 nonce/versioned control-directory bridge；extension 计算，adapter 转发。 |
| HIGH | current F2 registry 没有 External profile 过滤；Pi 专属设置会显示在普通 Grok。 | 先增加 external-only settings gate。 |
| HIGH | F2 built-in tools extension 只让 `--tools` 优先，`--no-builtin-tools` 可被 session_start 的 `setActiveTools()` 重新启用。 | 修正依赖 Issue 后再接入 rollback 开关。 |
| HIGH | 持久快照可能复制 cwd 外敏感内容且无限增长；现有 session 删除没有 sidecar 事件桥。 | 强制私有权限、byte blob 去重、显式清理/TTL/quota 和写前 fail-closed。 |
| MEDIUM | compaction 被误写为可能压缩掉 tree 历史。 | 已更正：Pi compaction 追加 entry，不删除原 session entries。 |

当前 Issue 在以上 Phase 1 未完成项关闭前不得进入实现。

## Control-Directory Bridge 契约

### 目录结构

```text
$GROK_PI_ROLLBACK_CONTROL/          # process-unique, 0700
  request-<nonce>.json              # adapter → extension
  response-<nonce>.json             # extension → adapter (atomic rename)
```

组合 binary 在启动时创建 `$GROK_HOME/pi-file-rollback/control-<pid>-<random>/`（0700），通过 child-only env `GROK_PI_ROLLBACK_CONTROL` 传给 Pi extension，同时把同一路径传给 `PiAgent`。进程退出时 best-effort 删除。

### 请求格式

```json
{
  "version": 1,
  "nonce": "<uuid-v4>",
  "sessionId": "<pi-session-id>",
  "method": "preview" | "execute",
  "params": {
    "targetEntryId": "<tree-entry-id>"
  },
  "createdAt": "<ISO-8601>"
}
```

### 响应格式

```json
{
  "version": 1,
  "nonce": "<same-uuid>",
  "sessionId": "<pi-session-id>",
  "method": "preview" | "execute",
  "ok": true | false,
  "result": {
    "eligible": true,
    "paths": [
      {
        "canonicalPath": "/abs/path",
        "action": "restore" | "delete" | "noop",
        "currentDigest": "sha256:...",
        "targetDigest": "sha256:... | null"
      }
    ],
    "conflicts": [],
    "transactionId": "<uuid> | null"
  },
  "error": "<string> | null",
  "completedAt": "<ISO-8601>"
}
```

### 安全与生命周期

- 所有文件权限 0600，目录 0700。
- 响应写入 `response-<nonce>.json.tmp` 后 `fs.renameSync()` 原子替换。
- Adapter 轮询间隔 50ms，超时 10s；超时后删除 request 并返回超时错误。
- Extension 处理前校验 nonce、sessionId、version；不匹配则忽略。
- 启动时清理超过 60s 的 stale request/response 文件。
- Adapter 不读写目标文件，不执行恢复；只转发请求和投影响应。

## Sidecar 清理与 Quota 契约

### 状态根

```text
$GROK_HOME/pi-file-rollback/        # 0700
  sessions/
    <session-id>/
      journal.jsonl                 # append-only MutationRecord + RollbackTransaction
      header.json                   # RollbackJournalHeader
  blobs/
    <sha256-hex>                    # content-addressed byte blobs, 0600
  control-<pid>-<random>/           # process-unique bridge dir
```

### Quota 限制

| 维度 | 默认限制 | 超限行为 |
|---|---|---|
| 单 blob | 10 MB | 阻止该次 write/edit 工具（fail-closed） |
| 单 session journal | 200 MB | 阻止新 mutation 捕获，toast 警告 |
| 全局 blobs | 2 GB | 阻止新 mutation 捕获，toast 警告 |

### 清理命令

`grok-pi rollback clean [--older-than <duration>] [--session <id>] [--dry-run]`

- `--older-than`：删除最后修改时间超过指定时长的 session journal 和孤立 blob。默认 30 天。
- `--session`：只清理指定 session。
- `--dry-run`：只报告将删除的内容和释放空间。
- 清理只删除 journal/header/孤立 blob；不删除 Pi session 文件。
- 孤立 blob = 不被任何现存 journal 引用的 blob。

### TTL 与自动清理

- 每次 grok-pi 启动时，best-effort 清理超过 30 天的 session sidecar。
- 不依赖 Grok/Pi session picker 删除事件（当前无该桥）。
- 清理失败不阻止启动。
