# Issue: Grok-Pi /resume 会话显示对齐

## Goal

在 Grok Pager 的原生 `SessionPicker` 中投影 Pi 会话的重命名状态与会话元数据，使 `/resume` 可辨识、可排序且不改变 Pi 对会话文件和切换的所有权。

## 背景/问题

当前 adapter 只投影 Pi JSONL 的标题、路径、起止时间和消息数；`session_info` 中的重命名状态被折叠进标题，且未提供模型、token/cost 等会话摘要。Pager 也无法区分重命名标题与首条消息，导致 `/resume` 的信息密度低于 Pi 原生选择器。

## 验收标准 (Acceptance Criteria)

- [x] Pi `session_info.name` 存在时，`/resume` 行以原生 Pager 样式明确标记为已重命名，而不是把名称与首条消息混淆。
- [x] 展开 Pi 会话行时显示绝对工作路径、开始时间、最近更新时间、模型、消息数和已记录的 token/cost；缺失字段不伪造数值。
- [x] 当前目录与全部目录的 Pi 会话均按最近活动时间倒序，重命名行的醒目显示不改变排序。
- [x] 不改变 Pi JSONL 写入、会话切换或 Pager 的原生渲染边界；adapter 保持 headless。
- [x] 补充定向单元测试并执行 adapter 测试与 `grok-pi` binary check（Pager lib 定向测试仍由既有五个无关编译错误阻断）。

## 关键决策

| 决策 | 理由 |
|------|------|
| 使用现有 `pi/ui/session_catalog` 通知 | Pager 已有原生 `SessionPicker` 接缝，避免新建 UI 或协议。 |
| adapter 只读扫描 JSONL | Pi 持有 session 文件格式和切换语义；adapter 不写文件。 |
| 缺失的 usage/cost 显示为缺省 | 不能从缺少数据的历史会话推导或伪造消耗。 |
