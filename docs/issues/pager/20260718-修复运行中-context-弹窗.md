# Issue: 修复运行中 Context 延迟并改为原生弹窗

## 用户症状

- 点击右上角 Context / `/context` 时，在 agent 运行中不会立即显示。
- 当前 turn 完成后，点击次数对应的 `ContextInfoBlock` 被批量追加到 scrollback。
- Context 是查看当前快照，不应污染对话历史。

## 根因

`handle_session_info()` 为取得 system/tool 分项调用 `__pi_context_breakdown`。该路径复用了 `run_bridge_command()`：它把 bridge 请求加入 `active_prompts`，并等待 `agent_settled`。运行中当前 turn 的 `agent_start` 会将该 bridge 标记为 `agent_started`，使 40ms 的 non-agent probe 失效，直到当前 agent settled 才返回。

## 设计

| 关注点 | 改动 |
|---|---|
| 运行中 snapshot | `__pi_context_breakdown` 改走专用 read-only bridge：仅请求 Pi RPC `prompt`，不进入 `active_prompts` / 不等待 `agent_settled`。Pi 的 extension command 在 streaming 时立即执行。 |
| 呈现 | `ContextInfoComplete` 不再 push `RenderBlock::ContextInfo`；更新 agent context 缓存后，打开 Pager 原生 `ModalWindow`，直接复用 `ContextInfoBlock` 的图形化 renderer。 |
| 重复点击 | 每个新快照替换同一 active modal，不写 scrollback，故不累计。 |
| 错误 | session/info 失败只显示原生 toast，不写 scrollback。 |

## 边界

- 可见 UI 完全由 `xai-grok-pager` 原生 `ModalWindow` / `DocViewer` 创建。
- adapter 仍 headless；不引入 Ratatui/Crossterm。
- 不改 Pi RPC/源码。
